1. Overview

TradeXV2 is a broker-agnostic algorithmic trading framework for Indian exchanges. It emphasizes process safety, canonical domain models, deterministic production behavior, and multi-adapter extensibility. The Python implementation mirrors its Java sibling (Trade_J) while retaining idiomatic Python semantics: dataclass-first domain types, threading.RLock for concurrency, and explicit resilience middleware.

Objective

Execute algorithmic strategies across brokers with standardized interfaces, risk enforcement, and replay-capable observability.

Scope

Multi-broker adapters, OMS, portfolio and risk, event bus, CLI/TUI diagnostics, datalake, analytics, and replay backtest.

Maturity

Production-oriented implementation validated by contract suites, chaos tests, HTTP healthz/readyz/metrics, and deterministic kill-switch behavior.

2. Design Principles

Canonical Domain-First

All broker adapters translate external payloads into canonical domain objects. Domain types are broker-agnostic and frozen dataclasses with slots, ensuring immutability, cheap allocation, and a single vocabulary across the system.

Capability-Based Composition

Broker capabilities are discovered at runtime via Capability enum registration. Consumers ask whether a connection supports a capability and retrieve provider instances dynamically, avoiding brittle RTTI or duplicated facade methods.

Resilience as Middleware

Rate limiting, circuit breaking, retry with backoff, and idempotency are applied uniformly via RetryExecutor pipelines. Cross-cutting concerns remain consistent and auditable across adapters.

Lifecycle Ownership

The LifecycleManager owns background services including websocket threads, token refresh schedulers, reconciliation, and daily PnL reset. close() drains all work to prevent leaked daemon threads.

Observability by Default

Production wire-up exposes /healthz, /readyz, and /metrics. Event metrics, HTTP observability, chaos suites, and kill-switch state give operators deterministic insight into system health.

Testability and Contract Compliance

Broker adapters are validated against reusable contract suites. Modules own their unit tests, contract tests, and integration markers. Chaos and determinism tests certify recovery and safety invariants.

3.1 Layer Contracts

Presentation Layer

The CLI/TUI converts user intent into service calls. It does not contain domain logic. Outputs include ASCII tables, Textual dashboard widgets, and JSON for scripting. Observability endpoints are served from this layer.

Service Layer

BrokerService owns the LifecycleManager, broker initialization, HTTP server startup, and wiring of OMS components. It is the single entrypoint into the running system. OmsService exposes OMS diagnostic views.

OMS Layer

OrderManager tracks in-flight and accepted orders, enforces idempotency, and coordinates risk checks before submission. PositionManager computes net exposure per instrument and segment. RiskManager evaluates ordered risk gates with RLock-protected atomicity.

Adapter Layer

Each adapter translates external protocol into canonical domain. It also registers SPI providers on the BrokerConnection. Adapters do not expose broker-native models outside their boundary except through canonical types and canonical DataFrame schemas.

Common Layer

Contracts ensure adapter compliance. Resilience policies are applied uniformly. The event bus delivers domain events safely. LifecycleManager guarantees deterministic startup and shutdown. Observability captures operational metrics.

Data Layer

Datalake captures market and portfolio state in a WAL-backed SQLite journal, schema-enforced DataFrames, and duckdb-backed analytics. Research and replay derive insight without mutating production state.

The system enforces behavioral contracts through abstract base classes and parameterized test suites, not convention alone.

The Broker ABC is the canonical interface. Every adapter completes the same surface, and the CLI commands operate against this abstraction rather than vendor specifics.

4.2 Capability-Based Connection

conn.register_capability(Capability.MARKET_DATA, MarketDataProvider(...))
conn.register_capability(Capability.ORDER_COMMAND, OrderCommand(...))

assert conn.has_capability(Capability.MARKET_DATA)
provider = conn.get_capability(Capability.MARKET_DATA)

This keeps the facade from hardcoding capability discovery. The CLI and diagnostic commands introspect supported capabilities for runtime feedback.

4.4 GatewayResult Monad

result = GatewayResult.success(data, metadata=meta)
        .map(transform)
        .flat_map(more_work)
        .recover(fallback)

GatewayResult wraps successful payloads with latency, source, and cache metadata, as well as failures with context. It avoids nested try/except and keeps pipelines composable.

4.6 Canonical DataFrame Schemas

Market data surfaces canonical pandas schemas:

  • HistoricalSchema: timestamp, open, high, low, close, volume, oi, symbol, exchange, timeframe
  • QuoteSchema: symbol, exchange, ltp, bid, ask, volume, oi, timestamp
  • OptionChainSchema: 16 columns including delta, gamma, theta, vega, rho
  • MarketDepthSchema: 20 bid and ask levels with size and orders

Broker-specific columns such as security_id are confined to adapter internals and are not leaked through public returns.

6. Broker Adapter Design

Adapters translate between vendor-specific protocols and the canonical domain. They also implement resilience and auth concerns close to the network boundary.

6.1 Dhan Adapter

Authentication

DhanAuthClient handles TOTP generation, token acquisition, and persistence. Modes include STATIC, TOTP_GENERATED, and WEB_RENEWABLE. Token refresh runs inside LifecycleManager.

Market Data

Historical, quote, depth, and option chain mapped via builders into canonical DataFrames with strict schema validation. Multi-bucket rate limiting isolates quotes from historical requests.

Orders & Portfolio

Order clients post requests and map statuses to canonical OrderStatus. Portfolio providers return positions, holdings, fund limits, and trades.

WebSockets

DhanMarketFeed, DhanOrderStream, and PollingMarketFeed are ManagedService instances, registered for lifecycle-managed startup and drain on close.

Reconciliation

ReconciliationService compares broker order state with OMS state, logs drift into journal, and triggers alerting when mismatched.

Circuit Breaker Split

Read, write, and auth circuit breakers are independent. Read failures no longer block order placement, improving live trading resilience.

6.2 Upstox Adapter

Upstox follows the same contract model with Capability-driven provider registration. It exposes market data v2/v3, orders, portfolio, margin, kill switch, alerts, GTT, cover/slice orders, and websocket v3 market data. Capabilities allow the router and diagnostics to describe supported features at runtime.

6.3 Paper Adapter

The paper broker is an in-memory simulator used for CI and strategy validation. It applies the same canonical interfaces and risk manager gates with deterministic fills and simulated latency.

7. Resilience and Safety

Reliability properties are enforced via reusable policies applied near the network boundary and coordinated through lifecycle-managed services.

Rate Limiter

TokenBucketRateLimiter provides token-bucket concurrency control with burst capacity. MultiBucketRateLimiter isolates rate limits per concern: orders, quotes, data, auth.

Circuit Breaker

States transition CLOSED → OPEN → HALF_OPEN with configurable thresholds. Separate circuit breakers for read, write, and auth prevent cascading failures and allow partial degradation.

Retry Executor

Each call passes through circuit breaker check, token acquisition, execution, and result handling. RetryableError triggers backoff with jitter. NonRetryableError fails fast to avoid load amplification.

Backoff and Jitter

Supports NoBackoff, FixedBackoff, ExponentialBackoff, and decorrelated jitter. Prevents thundering herd during recovery.

Risk Enforcement

  • Kill Switch: Deterministic flip blocks all placements immediately and is surfaced through /readyz and CLI diagnostics.
  • Position Limits: position_pct, gross_exposure_pct, and daily_loss_pct evaluated under RLock.
  • Daily PnL Reset: DailyPnlResetScheduler clears running PnL at IST 00:00.
  • Thread Safety: Every mutator and reader in RiskManager and PositionManager is atomic; raises cannot be interrupted mid-check to produce torn state.
  • Idempotency: Correlation-id backed caches and order checks prevent duplicate submissions under retry.

Chaos and Recovery

Ten deterministic failure-mode tests certify behavior under token expiry, kill-switch flips, concurrent order placement, idempotency under retry, lifecycle drain, and reconciliation drift. Chaos tests run without external dependencies and certify safety invariants without destructive side effects.

10. Quality Model

Quality is enforced across static analysis, typing, dependency checks, unit/contract/integration/chaos tests, and replay determinism verification.

Ruff Lint + Format

Ruff enforces E, W, F, I, B, UP, G, N, RUF, S, C4, SIM, and TID rules with project-wide per-file ignores for tests and scripts. Formatting checks ensure single-styled output.

Mypy Strictness

Type checking rejects incomplete definitions, unused ignores, casts, and partial returns. External broker SDKs and test directories are excluded with explicit overrides.

Contract + Chaos Suites

Parameterized contract tests verify adapter compliance. Deterministic chaos tests validate failure modes. E2E tests verify order lifecycle in controlled environments.

Replay Determinism

scripts/verify_event_replay confirms that replaying WAL events produces identical outcomes. This certifies data integrity for reconciliation, audit, and replay features.

Pre-commit and CI

Pre-commit hooks enforce formatting, linting, and test markers at commit time. CI pipelines gate merges on unit, contract, and selected integration test categories.

Security Scanning

Bandit and pip-audit are configured as dev dependencies and integrated into review workflows for dependency and implementation issue detection.

11. Architecture Review

The review examines current design choices, risks, strengths, gaps, and recommended improvements across structure, interfaces, operational behavior, and data model.

11.1 Strengths

  • Strong separation of concerns through canonical domain types and SPI ports reduces vendor lock-in.
  • Capability-driven composition and adapter registration allow extensibility without facade sprawl.
  • Lifecycle ownership eliminates leaked background threads and improves orderly shutdown behavior.
  • Resilience is applied uniformly via RetryExecutor, reducing adapter inconsistency.
  • HTTP observability and deterministic kill-switch behavior provide operator trust.
  • Contract and chaos suites verify behavior invariants beyond superficial unit coverage.

11.2 Risks and Gaps

  • Dual model system. Legacy Pydantic models coexist with canonical dataclasses. Boundaries are enforced by convention; gradual elimination is recommended to reduce cognitive load.
  • Dhan adapter size. DhanBroker is a large facade. Consider decomposing into capability composition per SPI port to reduce merge risk and improve testability.
  • Circuit breaker semantics. Although split by category, failure handling may still require clearer priority and operator controls for partial degradation.
  • TUI widget integration. Textual TUI and CLI services currently appear as separate surfaces; tighter integration through service clients could reduce command divergence.
  • Instrument resolver caching. Large CSV master files and caching behavior require memory profiling and eviction policy review for long-running sessions.
  • Coverage gaps. Current ruff violations include security-related rules. Hardcoded strings and broad exception handling may unintentionally mask credential or reliability issues.
  • Readiness semantics. /readyz currently depends on service state, but lacks caller-provided dependency checks such as exchange connectivity liveness.

11.3 Recommended Improvements

  1. Consolidate model systems. Deprecate legacy Pydantic models and route adapters fully through canonical dataclasses and canonical schema builders.
  2. Decompose DhanBroker. Reframe adapter as a collection of SPI port providers rather than a monolithic facade, matching Upstox and Paper patterns more closely.
  3. Operational policy config. Introduce declarative resilience profiles for read, write, and auth circuits so operators can tune thresholds without changing code.
  4. Unified interfaces between TUI and CLI services. Share backend clients to ensure consistent behavior and reduce command drift.
  5. Instrument catalog lifecycle. Add automatic periodic refresh, checksum validation, and least-recently-used eviction bounds.
  6. Address security lint findings. Resolve hardcoded strings, bare excepts, and logging format issues to reduce trust debt.
  7. Readiness expansion. Add optional external dependency checks into /readyz so orchestrators can distinguish application-up from able-to-trade.
  8. Schema governance. Enforce downstream adapter column contracts with typed schema validation at adapter boundaries.

11.5 Conclusion

TradeXV2 exhibits a mature, production-oriented design emphasizing canonical contracts, observable state transitions, and operator safety. The proposed improvements focus on consolidating dual model systems, reducing adapter complexity, and tightening operational semantics without altering the existing tested behavior. Maintaining backward compatibility with CLI, contract suites, and datalake formats should guide any refactoring.
